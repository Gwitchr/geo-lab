# Data Integrity

The invariants that must not break. Each one has a number attached because
each one was actually measured (see [[dev-notes]]'s verification sweep). If
you change the [[pipeline]] and one of these numbers moves, that is a
regression until proven otherwise â€” not a new baseline.

## 1. The generator is byte-reproducible

`data/gen/gen_listings.py` uses a fixed seed. Re-running it must produce
**byte-identical** `data/raw/listings_*.csv`. Verified by hashing two
consecutive runs and diffing:

```
$ python3 data/gen/gen_listings.py && sha256sum data/raw/*.csv > /tmp/a.txt
$ python3 data/gen/gen_listings.py && sha256sum data/raw/*.csv > /tmp/b.txt
$ diff /tmp/a.txt /tmp/b.txt && echo BYTE IDENTICAL
BYTE IDENTICAL
```

Everything downstream â€” the drop count, the duplicate count, the assignment
rate below â€” is only meaningful because this holds. A change to the
generator's RNG call order breaks all of it at once.

## 2. fastgeo parity passes on both backends

The parity suite (`pipeline/tests/test_parity.py`, seed `20260710`) must pass
**compiled** (13 tests: boundary points, degenerate rings, `batch_assign`
edge cases, `haversine_matrix`, `simhash64`/`hamming`), and the full pipeline
suite must pass under **`FASTGEO_FORCE_FALLBACK=1`** (51 passed, 1 skipped â€”
parity itself skips by design, since both sides would then be the fallback).
`pipeline/native/fallback/fastgeo_py.py` is the **spec**: a parity failure is
a `fastgeo` bug, never a reference bug. `-ffp-contract=off` in `CMakeLists.txt`
keeps boundary-point parity holding on arm64 â€” do not remove it, do not add
`-ffast-math`. See [[fastgeo]] and [[TESTING]].

## 3. AGEB assignment rate stays >= 95%

`enrich.py` currently assigns **50,297 / 50,314 CDMX listings = 99.97%**.
`ASSIGNMENT_TARGET = 0.95` is enforced in the script itself (it prints a
WARNING to stderr below the threshold). The 17 misses are understood, not
mysterious: they are near-duplicate rows whose coordinates were jittered
Â±0.0006Â° (~90 m) into a genuine gap between AGEB polygons, or just past a
boundary into an alcaldĂ­a `enrich.py` does not search (it only searches the
listing's own recorded `municipio`). `assign_ageb` correctly returns `None`
for a point outside every candidate polygon. A drop toward 95% means the
pre-filter or the polygon set changed â€” investigate, do not re-baseline.

## 4. `clean.py` drops exactly 750 rows â€” zero false drops

The generator injects exactly 750 intentionally-broken rows. `ingest.py`
reports:

```
read 100750 raw rows, kept 100000, dropped 750
{'bad_type': 162, 'bad_price': 281, 'missing_location': 152, 'bad_m2': 155}
```

Zero false drops, zero broken rows slipped through, and the per-reason
breakdown is part of the invariant â€” a total of 750 with a different
breakdown is still a regression.

## 6. `dup_of` is a LABEL, not a delete

`dedupe.py` sets `dup_of` (earliest `listed_date` in a cluster is canonical)
and **never deletes a row**. All 100,000 cleaned rows stay in `listings`.
Each consumer decides:

| consumer | excludes `dup_of IS NOT NULL`? |
|---|---|
| `pipeline/stats.py` | **yes** â€” every aggregate filters `WHERE dup_of IS NULL` |
| `pipeline/export_web.py` | **yes** â€” `WHERE dup_of IS NULL`, hence 97,508 in-scope rows before the 5,000 cap |
| `geoq` (`query` / `stats` / `export`) | **no** â€” reads the raw table, so its results and counts include the 2,492 flagged rows |

This asymmetry is **known and deliberate**, not a bug. Making dedup
table-wide is a [[query-engine]]-side decision that has not been made. Anyone
comparing a `geoq` count against a `stats.json` count must know this â€” if you
change it, change it on purpose and update both sides plus [[contracts]].

## 7. Synthetic data is never labelled as real

`data/raw/` and `data/db.sqlite` are **synthetic** â€” generated by
`gen_listings.py`, realistic in distribution (real place names, price
gradients derived from the real marginaciĂłn index for CDMX) but corresponding
to no real property. The only real data is `data/geo/agebs_cdmx.geojson`
(INEGI) and `data/geo/marginacion_cdmx.csv` (CONAPO), both credited with
source URLs, retrieval dates, and licenses.

**Synthetic data must NEVER be labelled as INEGI/CONAPO-sourced.** No file
under `data/` may be a placeholder wearing a real dataset's filename without
an explicit placeholder label. The Guadalajara/Monterrey price baselines and
colonia gazetteers are hand-curated approximations and must keep saying so â€”
unlike CDMX's, they are not backed by an open dataset. See [[data-notes]].
