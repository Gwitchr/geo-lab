# geo-lab

A personal "geo lab": start practical — collect and explore property listings —
and grow it into open-data geographic analysis (census areas, marginación
index, business directory) to answer questions like "¿dónde en la CDMX
funcionaría un concierto mediano?" or how marginalization maps onto price per
m². Mexico is the first region, kept region-friendly rather than hardcoded.

Four pieces: a Python data pipeline, a TypeScript CLI (`geoq`) for querying the
listings database, a Vite + TypeScript web explorer (table, charts, map), and
`fastgeo`, a C++17/pybind11 module for the spatial hot loops (point-in-polygon
AGEB assignment, near-duplicate detection).

## Quickstart

`data/db.sqlite` (built artifact, ~100k synthetic listings) and
`web/public/listings.json` (the real export of it, capped at 5,000 rows) are
committed, so the web app and CLI both work straight off a clone with no
Python setup required.

```sh
git clone <this repo> && cd geo-lab

# Web explorer
cd web
npm ci
npm run dev      # http://localhost:5173 — table, charts, map over real data
```

In another terminal:

```sh
cd cli
npm ci
npm run build
node dist/index.js query "price<2500000 and colonia:roma" --db ../data/db.sqlite
```

(or skip the build and use `npm run dev -- query "..." --db ../data/db.sqlite`,
which runs the TypeScript directly via `tsx`.)

Both of the above were re-verified from a clean copy of this tree (fresh
`npm ci`, no pre-existing `node_modules`) as part of preparing this README —
see `docs/dev-notes.md`'s final verification sweep for the transcripts.

### Regenerating the data from scratch (optional)

Everything under `data/` is reproducible from `data/gen/gen_listings.py`'s
fixed seed. Only needed if you want to touch the pipeline itself; the
committed `data/db.sqlite` already reflects a full run of this:

```sh
python3 -m venv .venv-pipeline && source .venv-pipeline/bin/activate
pip install pytest              # the only dev dependency; pipeline/*.py is stdlib-only

python3 data/gen/gen_listings.py    # -> data/raw/listings_*.csv (~100k rows)
python3 pipeline/ingest.py          # -> data/db.sqlite
python3 pipeline/enrich.py          # AGEB + marginación assignment for CDMX rows
python3 pipeline/stats.py           # -> pipeline/stats.json
python3 pipeline/export_web.py      # -> web/public/listings.json (capped at 5000)

cd pipeline && python3 -m pytest -q
```

The C++ `fastgeo` module is optional — `pipeline/geo_backend.py` falls back to
a pure-Python implementation automatically when the compiled extension isn't
installed, so all of the above works without a C++ toolchain. To build it:

```sh
python3 -m venv .venv-native && source .venv-native/bin/activate
pip install pybind11 cmake ninja scikit-build-core pytest numpy
pip install ./pipeline/native
cd pipeline && python3 -m pytest tests/test_parity.py -v   # compiled vs. Python reference
```

## Stack

TypeScript (web + CLI), Python 3.11+ (data pipeline), C++17 via pybind11 for
the spatial hot loops. Kept deliberately small: vitest/pytest for tests, plain
eslint, no frameworks beyond what a page and a CLI need.

## Roadmap

Three layers, built in order:

1. **Listings explorer** (done) — collect, clean, dedupe, and browse property
   listings: `geoq` for querying, the web app for a table/filter/charts/map
   view.
2. **Geo enrichment** (done for CDMX) — assign listings to their INEGI AGEB
   polygon and CONAPO marginación grade, so location can be analyzed against
   real socioeconomic geography, not just raw lat/lng.
3. **Socio-cultural analysis** (exploring) — the open-ended layer: business
   directory overlays, event-venue siting questions ("where would a mid-size
   concert work"), price-vs-marginación correlation across the metro. No
   fixed scope yet; this is where the "lab" part of geo-lab lives.

## Data sources

`data/geo/agebs_cdmx.geojson` (INEGI Marco Geoestadístico urban AGEB polygons
for CDMX, via the CDMX open-data portal) and `data/geo/marginacion_cdmx.csv`
(CONAPO's Índice de Marginación Urbana 2020, filtered to those AGEBs) are real,
credited open data — exact source URLs, retrieval dates, and licenses are in
[`docs/data-notes.md`](./docs/data-notes.md), along with how the CDMX colonia
gazetteer (IECM 2019) and the Guadalajara/Monterrey curated gazetteers were
built. Everything under `data/raw/` and `data/db.sqlite` is **synthetic**
listings data generated by `data/gen/gen_listings.py` — realistic in
distribution (real place names, price gradients derived from the real
marginación index for CDMX) but not scraped from any real listing site.

## Why fastgeo exists

The pure-Python spatial join is the real bottleneck at scale. Measured on this
machine (macOS arm64, AppleClang, `-O3 -ffp-contract=off`, single-threaded),
same fixture, same inputs, both implementations verified to agree bit-for-bit:

- **`batch_assign`, 100,000 points × 2,000 polygons**: compiled `fastgeo` 2.19s
  vs. the pure-Python fallback 247.32s (4 min 7s) — a **~113x** speedup.
- **`dedupe.py`'s simhash near-duplicate pass, full 100,000-row run**: compiled
  5.3s vs. fallback 22.5s — a **~4.3x** speedup in the actual pipeline
  workload (smaller than the raw `batch_assign` number because `enrich.py`
  pre-filters candidate polygons before ever calling into `fastgeo`, while
  `dedupe.py`'s simhash-every-description workload has no equivalent cheap
  pre-filter to shrink the gap).

Both numbers, and the calibration work behind `dedupe.py`'s match thresholds,
are in [`docs/dev-notes.md`](./docs/dev-notes.md).
